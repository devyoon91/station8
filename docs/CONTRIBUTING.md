# CONTRIBUTING

이 문서는 ESP API 프로젝트의 브랜치 전략, 커밋 컨벤션, PR 절차 등 '협업 규정'을 정의합니다.

## 1. 브랜치 전략

- **main**: 제품 출시 및 배포 브랜치.
- **develop**: 다음 릴리스를 위한 개발 브랜치.
- **feature/<issue-number>-<description>**: 개별 기능 개발 브랜치.

## 2. 커밋 컨벤션

커밋 메시지는 아래 형식을 준수하며, 제목은 영문(명령조, 50자 이내), 본문은 한글로 작성합니다.

### 키워드 (Type)

- `FEATURE`: 주요 기능 추가 또는 수정
- `FIX`: 버그 해결
- `DOCS`: 문서 추가, 삭제, 변경
- `REFACTOR`: 로직 변경 없는 코드 구조 개선
- `TEST`: 테스트 코드 추가/수정
- `CHORE`: 빌드 업무, 패키지 매니저 설정 등

### 작성 규칙

- **제목**: 첫 글자 대문자, 끝에 마침표(`.`) 금지, 명령조로 작성.
- **본문**: 무엇을/왜 변경했는지 한글로 상세히 기술.
- **형식**: 제목과 본문은 한 줄 띄워 분리하며, 본문은 한 줄당 72자 내로 직접 줄바꿈 처리.

### 예시

```
FEATURE: 로그인 기능 추가

- 로그인 서비스 및 API 추가
- password SHA256 암호화 적용
```

---

## 3. Pull Request 절차

1. `develop` 브랜치에서 기능 브랜치를 생성합니다.
2. 작업 완료 후 `develop` 브랜치로 PR을 생성합니다.
3. 최소 1인 이상의 코드 리뷰 및 승인(Approve) 후 머지합니다.
4. CI 빌드 및 테스트 통과를 반드시 확인합니다.
